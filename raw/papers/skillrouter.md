# SkillRouter: Skill Routing for LLM Agents at Scale

**论文信息**

- 标题: SkillRouter: Skill Routing for LLM Agents at Scale

---

## 1. 这篇论文到底在做什么？

解决大规模 LLM Agent 技能路由问题：当技能库扩展到数万个可复用技能时，如何从用户任务精准定位最相关的技能，同时保证推理效率。

核心矛盾：现有 agent 系统采用 Progressive Disclosure（只暴露技能名称/描述，隐藏实现体），但这导致路由精度大幅下降。

---

## 2. 为什么要做这个？（研究动机）

**背景：** LLM Agent 的技能生态正在快速扩张——Claude Skill Registry、OpenClaw Skills 等系统已积累数千至数万个技能。每一轮用户交互都需要从候选池中选出相关技能，但 inference 时无法加载全部技能的实现体。

**核心问题：** Progressive Disclosure 策略造成路由精度严重损失——在 ~80K 候选规模下，隐藏技能体导致精度下降 **31–44 个百分点**。这说明技能体全文是远比元数据更关键的路由信号。

**研究空白：** 现有方法（稀疏检索、密集检索、重排）均假设技能体不可见，没有系统在"技能体全文信号"和"推理效率"之间取得良好平衡。

---

## 3. 是怎么做到的？（技术路线）

### 3.1 关键发现（先行实验）

在 SkillsBench 基准（约 80K 候选技能）上，对比"暴露技能体"vs"隐藏技能体"两种设置：

| 基线方法 | 隐藏技能体 | 暴露技能体 | 精度损失 |
|---------|-----------|-----------|---------|
| BM25 | 基准 | +31-44 pp | 严重 |
| Dense | 基准 | +31-44 pp | 严重 |
| Reranker | 基准 | +31-44 pp | 严重 |

结论：**技能体全文是路由的关键信号**，而非边缘优化。

### 3.2 SkillRouter 架构：1.2B Retrieve-and-Rerank 管道

```
用户任务 → BM25 Sparse Retrieval（Top-K 召回）→ 1.2B Reranker（技能体重排）→ Hit@1 技能
```

**两阶段设计：**
1. **稀疏检索阶段**：BM25，从 80K 候选中快速召回 Top-K（约100-200）
2. **重排阶段**：1.2B Reranker，将技能体全文作为关键信号重新排序

**为什么这样设计：**
- 80K 规模直接做密集检索计算量太大
- BM25 稀疏检索快速缩小候选范围
- Reranker 用小模型（1.2B）捕捉技能体全文的深层语义

**训练数据：** 在 SkillsBench 上自动构建含 80K 技能的大规模路由训练集

### 3.3 与现有方法的对比

| 方法 | 技能体可见性 | 参数量 | 推理速度 |
|------|------------|--------|---------|
| 纯 BM25 | 隐藏 | - | 快 |
| 纯 Dense | 隐藏 | 大 | 慢 |
| SkillRouter | 暴露 | 1.2B | 快（5.8× vs 最强基线）|

---

## 4. 做得怎么样？（关键数据）

| 指标 | 数据 |
|------|------|
| **Hit@1**（SkillsBench, 80K 候选） | **74.0%**（最强平均 top-1 路由） |
| 精度提升（vs 隐藏技能体基线） | +31–44 pp |
| 参数量 | 1.2B（比最强基线少 **13×**） |
| 推理速度 | 比最强基线快 **5.8×** |
| 跨基准泛化 | 在独立构建的补充基准上同样有效 |
| 端到端 coding agents 研究 | 路由增益可转化为任务成功率提升，能力越强增益越大 |

---

## 5. 意味着什么？（研究意义）

1. **证明技能体全文是关键路由信号**：Progressive Disclosure 的精度损失是系统性的，不可忽视
2. **小模型能干大事**：1.2B Reranker 在特定任务上超越大模型，关键在于训练范式和信号选择
3. **端到端验证**：从 routing 精度到 coding agent 任务成功的完整链路，路由增益可传导
4. **Benchmark 贡献**：SkillsBench 填补了大规模技能路由评测的空白

---

## 6. 优点/亮点

1. **实验设计扎实**：80K 候选规模足够真实，对比基线全面（稀疏/密集/重排全覆盖）
2. **发现有价值**：31-44 pp 的精度损失是一个明确、可量化的系统性问题
3. **效率优势显著**：13× 参数减少 + 5.8× 加速，且精度更高，工程落地价值大
4. **端到端验证完整**：不只做 routing 评测，还验证了 routing → agent task success 的传导链

---

## 7. 潜在问题/局限

1. **评测基准来源**：SkillsBench 来自特定技能来源（SkillBench2026），泛化性需要在更多样化的技能集合上验证
2. **技能体质量依赖**：reranker 效果依赖技能体本身的质量和标准化程度；真实世界技能体质量参差不齐
3. **Top-K 召回依赖**：BM25 阶段如果召回不足，reranker 也无法补救；两阶段误差累积
4. **未见方案对比**：没有和 GPT-4o/Gemini 等大模型端到端做 routing 的对比
5. **未见中文技能场景**：论文主要基于英文技能集合，中文场景的技能路由效果未知

---

## 8. 未解决的关键问题

1. 如何处理技能体噪声/质量差异？（当前假设技能体是有质量的）
2. 如何动态更新技能库而不用重新训练 reranker？
3. 技能路由与 agent planning 的更深层联合优化是否存在？
4. 更大规模（>100K）技能库下的路由精度和效率权衡

---

## 9. 与你研究的关联

SkillRouter 与以下方向高度相关：

- **Agent 工作流 / 工具选择**：技能路由本质上是"工具选择"问题的扩展版本；全文信号比描述更关键的发现，对工具选择有直接借鉴
- **记忆/上下文管理**：类似地发现"全量信息"vs"摘要信息"的精度差异，与 AgentFold 的上下文压缩问题呼应
- **工具路由的效率优化**：小模型（1.2B）完成高精度路由，与 agent 工具调用轻量化趋势一致

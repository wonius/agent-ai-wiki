## 第二部分：一句话概括（≤50字）

**核心贡献**: 提出多LLM智能体框架，将工具学习能力分解为 planner/caller/summarizer 三个角色，通过两阶段微调使小模型也能高效完成复杂工具调用任务。

---

## 第三部分：科学问题

### 核心问题
小模型在工具学习（tool learning）任务中性能受限，无法同时掌握任务规划、工具调用和结果总结三项能力。

### 背景与重要性
- LLM 智能体能够调用外部工具（API、函数）完成复杂任务
- 传统方法训练单一 LLM 具备所有能力，但小模型效果差
- 工具学习需要：理解用户查询、生成答案、任务规划、工具调用、结果总结

### 现有方法不足
- 单一 LLM 需同时学习多种能力，导致性能瓶颈
- 小模型（7B以下）在工具学习任务上表现不佳
- 缺乏针对工具学习能力的专门训练范式

---

## 第四部分：技术路线

### 核心技术方案
1. **能力分解**: 将工具学习能力分解为三个独立组件
   - **Planner**: 任务规划，决定下一步做什么
   - **Caller**: 工具调用，执行具体工具
   - **Summarizer**: 结果总结，汇总工具返回结果

2. **两阶段训练范式**
   - **Stage 1**: 在整个数据集上微调 backbone LLM，提供全面任务理解
   - **Stage 2**: 在各自的子任务上分别微调 planner/caller/summarizer

3. **模块化框架优点**
   - 可独立更新各组件
   - 可使用更小的 LLM 构建各组件
   - 支持组件间的协作

## 第五部分：核心发现

### 主要结论
- 多 LLM 框架在各种工具学习基准上超越传统的单一 LLM 方法
- 框架的有效性和在工具学习中的优势得到验证
- 小模型通过分工协作可以达到甚至超越单一 大模型的性能

### 关键创新
- 首次将工具学习能力显式分解
- 提出可训练的三组件协作框架
- 两阶段训练策略确保各组件专业化

| 方法 | 特点 | 局限 |

| 本文方法 | 能力分解+协作 | 需要协调三个模型 |

## 第八部分：局限性

1. **组件间通信开销**: 三个组件间传递信息可能引入延迟
2. **协调复杂性**: 需要额外的协调机制确保组件正确协作
3. **错误传播**: 任何一个组件出错可能影响整体性能
4. **资源需求**: 虽然用小模型，但需要三个模型实例

## 第十部分：对Agent实践的指导

### 可直接借鉴的方法
1. **能力分解思路**: 将复杂任务分解为规划、执行、总结三个阶段
2. **协作框架设计**: 多 agent 协作而非单一 agent 承担所有责任
3. **专业化训练**: 先泛化再专业化是两阶段训练的有效策略

### 需要适配的场景
- 非工具调用场景：需要调整三个角色的职责
- 实时性要求高的场景：考虑组件间通信延迟
- 资源受限场景：可使用更小的专业化模型

### 潜在应用
- 复杂软件开发任务的多步骤规划
- 需要调用多个API的企业工作流自动化
- 教育领域的自适应辅导系统

## 一、研究背景与问题

### 1.1 Web Agent 的意义

让 LLM 直接操作网页完成信息搜索与综合，是现代 AI 最重要的应用方向之一。OpenAI Deep Research 等系统已经证明这类 Agent 在复杂信息获取任务上有巨大潜力。然而，长程任务（long-horizon tasks）仍是当前 Web Agent 的核心瓶颈。

### 1.2 现行方案的两大困境

当前主流的 Web Agent 大多基于 **ReAct 范式**（Yao et al., 2023）——在"推理→行动→观察"的循环中累积每一步的思考、动作和结果到上下文中。

这种方案在短任务上有效，但在长任务上面临一个**根本性的两难**：

**方案 A：持续追加历史（ReAct 模式）**
- 优点：信息完整性高，不丢失任何细节
- 缺点：上下文无限膨胀，噪声淹没关键信号，最终导致上下文饱和（context saturation）

**方案 B：每步全量摘要**
- 优点：上下文始终保持精炼
- 缺点：每次压缩都是对历史的"有损转录"，关键细节可能在某次摘要中被永久丢弃

论文给出了一个量化说明：如果假设每步全量摘要有 1% 的概率丢失关键细节，那么：
- 第 1 步发现的关键信息，**100 步后存活率只剩 36.6%**（0.99^100）
- **500 步后仅剩 0.66%**（0.99^500）

### 1.3 认知科学的启发

人脑是怎么处理这个问题的？论文引用了认知科学的经典结论：人不会"边做边忘"，也不会"每步重写全部记忆"。真正的机制是**回顾性整合（retrospective consolidation）**——在关键节点主动回顾：丢弃无关步骤、提炼中间发现、抽象关键洞察。

论文将这个认知过程形式化为 **"folding"（折叠）操作**，并由此提出 AgentFold 范式。

## 二、核心方法

### 2.1 总体架构：Perceive → Reason → Fold → Act 循环

AgentFold 的核心创新在于**把上下文管理从隐式副产品变成显式的、可学习的一步**。

每一步交互包含以下环节：

```
感知（Receive Context）
  → 推理（Generate Response = Thinking + Folding + Explanation + Action）
    → 执行 Action（获取 Observation）
      → 更新上下文（Folding 指令立即生效）
        → 循环
```

与 ReAct 的本质区别：ReAct 的循环是 `Reason → Act → Observe → Reason`，而 AgentFold 在 `Act` 之后增加了 `Fold` 这一显式步骤，让 Agent 主动决定如何管理自己的历史。

### 2.2 上下文结构：Ct = (Q, T, St-2, It-1)

AgentFold 的上下文由四个组件构成：

**① Q = 用户问题（Question）**
- 始终不变，作为整个任务的锚点

**② T = 可用工具列表（Tools）**
- 始终不变，包含每个工具的名称、描述和参数 schema

**③ St-2 = 多尺度状态摘要（Multi-Scale State Summaries）**
- 这是 AgentFold 最核心的创新
- 由若干"摘要块"组成的有序序列，每个块代表若干步的历史
- 形式化表示：St = (s(x1,y1), s(x2,y2), ..., s(xm,ym))
- 其中 s(x,y) 表示将第 x 到第 y 步（连续的）压缩为一段文本摘要
- 块与块之间step范围**首尾相接**，完整覆盖到上一步（t-2）
- **多尺度性质**：单步粒度凝缩记为 s(x,x)，跨步深度整合记为 s(x,y)（y > x）

**④ It-1 = 最新交互（Latest Interaction）**
- 上一轮的完整记录，包含三个部分：解释 et-1、动作 at-1、观察 ot-1
- It-1 = (et-1, at-1, ot-1)
- 提供最原始、最完整的近期细节，供 Agent 做短期决策

**为什么这样设计？**

 Latest Interaction 保证近期细节**完全不丢失**——做下一秒决策时，有完整的原始信息可用。同时，多尺度状态摘要提供**去噪后的全局视角**——让 Agent 不会被无关细节淹没，能保持连贯的长期规划能力。两者结合，直接破解了"细节保留"和"上下文精简"之间的两难。

### 2.3 AgentFold 的响应结构：Rt = (tht, ft, et, at)

每一步，AgentFold 生成一个四元组响应：

**① tht = 思考（Thinking）**
- Agent 的内部推理过程
- 不写入上下文，仅用于生成其他组件

**② ft = 折叠指令（Folding Directive）**
- 这是整个范式的核心
- 指令立即作用于上下文，将历史记录更新为新摘要块
- 两种操作模式（见下节）

**③ et = 解释（Explanation）**
- 对思考过程关键洞察的精炼总结
- 阐述选择当前行动的理由
- 构成下一轮的 It 的一部分

**④ at = 行动（Action）**
- 要么是工具调用（附工具名和参数）
- 要么是最终答案（当 Agent 认为已收集足够信息时）

### 2.4 两种折叠操作

这是 ft 的具体实现。AgentFold 有两种互补的折叠模式，Agent 在每步自行决定使用哪种：

**模式一：粒度凝缩（Granular Condensation）— k = t-1**

将 Latest Interaction（即 It-1 = et-1, at-1, ot-1）压缩为单个摘要块 s(t-1,t-1)，加入 St-1。

**效果：** 把上一整步的完整交互（可能很长）压缩成一条精炼陈述，保留关键结论，去除原始噪声。同时，该摘要块是**独立存在**的，后续的深度整合不会影响到它内部的内容——它已经是一个不可再分割的整体。

**模式二：深度整合（Deep Consolidation）— k < t-1**

将 Latest Interaction 与若干历史摘要块合并为一个粗粒度摘要 s(k, t-1)。

**效果：** 把多个步骤（可能跨越整个失败的子研究方向）彻底合并为一句结论，抹去中间的所有中间步骤和噪声。例如："[Compressed Step 6 to 16] 该方向不可行"。

### 2.5 Acting 与 Reflecting 的认知协同

这是论文一个重要的设计洞察。AgentFold 的响应结构将**制定下一步行动**和**管理自身上下文**紧密耦合：

- **制定行动**要求 Agent 必须审视近期历史，找到关键线索 → 这个过程自然产生了"什么值得保留在摘要中"的信号
- **制定折叠指令**要求 Agent 必须评估当前轨迹，提炼最关键信息 → 这个过程反过来让 Agent 更清晰地理解了当前状态

两者互相强化，形成一个**自我调节的正向循环**："行动→反思→更好行动→更精准的上下文管理"。

### 2.6 Fold-Generator：训练数据收集

由于不存在现成的、包含显式折叠操作的轨迹数据集，论文设计了一套数据收集流程 **Fold-Generator**：

**核心思路：** 用超大模型（开源的 Qwen3.5-72B-Instruct 等）作为"教师"，生成包含"推理→折叠→行动"完整轨迹的数据，然后用较小的专用 Agent 模型（30B）来学习。

**关键设计：** 数据收集时用更大规模的通用模型，训练时蒸馏到专用的、推理时更高效的 30B Agent 模型上。完全不依赖持续预训练，也不使用强化学习。

## 三、实验结果

### 3.1 主要结果（表1）

| 模型 | BrowseComp | BrowseComp-ZH | WideSearch | GAIA |
|------|-----------|--------------|-----------|------|
| **AgentFold-30B-A3B（本文）** | **36.2** | **47.3** | **62.1** | **67.0** |
| DeepSeek-V3.1-671B-A37B | 30.0 | 49.2 | — | 63.1 |
| GLM-4.5-355B-A32B | 26.4 | 37.5 | — | 66.0 |
| Kimi-K2-1T | 14.1 | 28.8 | 59.9 | 57.3 |
| OpenAI o4-mini | 28.3 | 44.3 | — | — |
| OpenAI o3 | 49.7 | 58.1 | 60.0 | 70.5 |
| OpenAI Deep Research | 51.5 | 42.9 | — | 67.4 |

**关键结论：**
- 在开源模型中全面领先，刷新 SOTA
- 30B 参数模型超越 671B 的 DeepSeek-V3.1（规模差距 20 倍以上）
- WideSearch 得分 62.1，超越所有闭源模型包括 o3 和 Claude-4-Sonnet

### 3.2 上下文 Token 数增长分析

实验从 BrowseComp 采样 200 条轨迹，分析每步的上下文 token 数量变化：

- **100 步后**：平均上下文仅约 **7k tokens**（模型 128k 上限的 5%）
- 增长模式：约从 3.5k **次线性增长**到 7k，远未触及上限
- 这意味着 AgentFold **还有大量空间可以继续扩展交互步数**

作为对比：ReAct 的上下文在 100 步后已经线性膨胀到难以承受的程度。

### 3.3 上下文块数量分析

"块"指的是多尺度状态摘要 St 中的一个摘要单元（可以是一步的凝缩，也可以是多步的整合），加上 Latest Interaction。

- **ReAct**：块数量随步数线性增长（每步+1个新块）
- **AgentFold**：块数量次线性增长，因为深度整合操作持续将多个块合并为 1 个

这说明 AgentFold 的上下文**不仅 token 少，结构也始终保持精简**。

### 3.4 深度整合的真实案例

论文附录 Case Study 1 记录了一个具体轨迹（表2），展示了深度整合的实际效果：

以第 19 步为例，此时 Agent 经历了 step 6 到 step 16 共 11 步的失败尝试（方向走不通）。AgentFold 在第 19 步执行了一次**深度整合**，将整个 11 步序列压缩为一句话：

> `[Compressed Step 6 to 16] 该方向不可行`

这一步操作的效果：
1. 上下文瞬间精简：11 步详细记录 → 1 个摘要块
2. 关键教训得以保留："这个方向不行"
3. 无用噪声彻底清除：中间所有无关操作记录消失
4. Agent 基于这个新整合的认知，重新规划并切换到新的搜索方向

### 3.5 交互步数 Scaling 特性

论文在 BrowseComp 上测试了不同交互步数上限下的表现，对比 AgentFold-30B 和 GLM-4.5-355B（规模大 10 倍以上）：

- GLM-4.5 基线在 **64 步之后性能饱和**，无法继续扩展（因为上下文膨胀到无法继续）
- AgentFold-30B 在 **256 步上限**的测试中仍然保持有效，且始终优于 355B 基线

这说明 AgentFold 的架构能够**随着允许的交互步数增加而持续提升性能**，不存在明显的天花板。

## 四、与现有工作的关系

### 4.1 Web Agent 发展脉络

| 阶段 | 代表工作 | 核心特征 |
|------|---------|---------|
| 早期探索 | OpenAI Deep Research | 端到端 web Agent 原型 |
| 主流范式 | ReAct (Yao et al., 2023) | 推理-行动-观察循环，append-only 上下文 |
| 长程优化 | WebThinker, WebDancer, WebSailor 等 | 在 ReAct 基础上改进，但仍为 append-only |
| **AgentFold（本文）** | — | **显式折叠操作，主动管理上下文** |

AgentFold 的改进直接针对 ReAct 的根本缺陷：上下文膨胀问题不是通过"加速"或"限制步数"来缓解，而是通过**改变上下文管理范式**从根本上解决。

### 4.2 上下文管理的两条路线

**外部上下文增强（External Context Augmentation）**
- 将外部知识（用户画像、历史对话等）注入上下文
- 代表：MEM0, A-Mem, Memory3 等
- AgentFold 的工作与此正交

**任务内上下文整理（Intra-Task Context Curation）**
- 管理任务本身产生的信息，而非引入外部信息
- 代表：MEM1, MemAgent（均为每步全量摘要）
- **AgentFold 与这些方法的关键区别：不是每步全量压缩，而是多尺度的、按需的折叠**

## 五、局限性

1. **折叠策略依赖模型自行判断**：没有外部信号保证折叠时机最优，模型可能误判何时该凝缩、何时该整合
2. **子任务完成判断的主观性**：Agent 自行决定"某子任务已完成"并触发深度整合，若判断错误会导致关键信息提前丢失
3. **评估范围有限**：仅在信息搜索类任务（BrowseComp、WideSearch、GAIA）上验证，对其他类型 Agent 任务（如代码生成、对话等）泛化性未知
4. **训练数据由更大模型生成**：存在专家模型的固有上限，且生成轨迹质量依赖大模型能力
5. **下一步方向（论文提及）**：使用强化学习（RL）让 Agent **自主发现**更优的折叠策略，而非依赖 SFT 模仿

## 六、一句话总结各部分

- **问题根源**：ReAct 的 append-only 上下文导致信息膨胀 vs 全量摘要导致信息不可逆丢失
- **核心洞察**：人脑在长程任务中使用"回顾性整合"——不边做边忘，也不每步重写，而是按需主动压缩
- **核心机制**：每步显式生成折叠指令（Granular Condensation / Deep Consolidation），将历史变为动态可管理的多尺度工作空间
- **关键结果**：30B 模型超越 671B 模型，100 步后上下文仅 7k tokens，支持 500+ 步交互
- **核心启示**：上下文管理本身就是一种能力——Agent 不仅要会执行，还要会管理自己的信息

## 七、与你的关联

**相关度：高**

你目前在做 Agent 工程，上下文管理是核心工程问题之一。这篇论文直接解决的就是"Agent 如何在长程任务中保持上下文质量"的问题。

**实际可借鉴之处：**

1. **多尺度摘要 + Latest Interaction 的结构**可以直接用于改进 ReAct 风格的 Agent，实现上比引入向量数据库简单得多
2. **按需凝缩而非每步全量压缩**的策略——如果你的 Agent 任务步数较多，这个模式值得作为默认选项
3. **Acting 和 Reflecting 耦合的设计**：在设计 Agent 的 prompt/输出结构时，显式要求 Agent 每步同时给出"行动理由"和"上下文管理决策"，两者互相强化

**需要验证的点：**
- 折叠操作的有效性高度依赖模型的 SFT 训练质量，直接迁移到其他模型可能效果打折
- 论文的方法在信息搜索类任务上效果显著，在其他类型任务上需要进一步验证

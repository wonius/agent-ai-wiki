## 2. 一句话概括

提出AgentReuse计划复用机制，利用意图分类评估请求相似度，有效复用已有计划，降低延迟93.12%。

---

## 3. 科学问题

### 核心问题
如何减少LLM驱动Agent的计划生成延迟，提升用户体验？

### 背景与重要性
- LLM驱动Agent的计划生成延迟可达数十秒
- 实际数据分析：约30%请求是相同或相似的
- 传统语义相似度方法无法准确判断可复用性

### 现有方法不足
- 传统语义缓存（如GPTCache）直接比较原始文本
- 不同参数但同类型请求被误判为不相似
- 计划文本非结构化，难以有效复用

## 4. 技术路线

### 核心技术方案：AgentReuse 计划复用机制
```
┌─────────────────────────────────────────────────────────────────────┐
│                       AgentReuse 流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                      │
│   User Request (Request 2)                                           │
│   "Book a ticket from Changsha to Shanghai for tomorrow"            │
│       │                                                              │
│       ▼                                                              │
│   ┌────────────────────────────────────────────────────────────┐      │
│   │  1. Key Parameter Identification (关键参数识别)              │      │
│   │     • 语义差异分析                                        │      │
│   │     • 提取关键参数: 时间、起点、终点                        │      │
│   │     • Time: tomorrow / Origin: Changsha / Dest: Shanghai   │      │
│   └──────────────────────────────────────────────────────────────┘      │
│       │                                                              │
│       ▼                                                              │
│   ┌────────────────────────────────────────────────────────────┐      │
│   │  2. Intent Classification-based Similarity (意图分类)      │      │
│   │     • 判断请求类型是否相同                                 │      │
│   │     • 同类型 → 计划可复用                                 │      │
│   │     • 语义相似度 + 意图分类                               │      │
│   └──────────────────────────────────────────────────────────────┘      │
│       │                                                              │
│       ▼                                                              │
│   ┌────────────────────────────────────────────────────────────┐      │
│   │  3. Plan Structured Conversion (计划结构化)               │      │
│   │     • 转换为结构化执行计划                                 │      │
│   │     • 替换关键参数                                         │      │
│   │     • 返回可执行计划                                      │      │
│   └──────────────────────────────────────────────────────────────┘      │
│       │                                                              │
│       ▼                                                              │
│   Result: 复用 Request 1 的计划，替换关键参数                       │
└─────────────────────────────────────────────────────────────────────┘
```

### 关键创新

**1. 关键参数识别**
- 利用语义差异识别关键参数
- 解决同义词、不同语序问题
- 参数：时间、起点、终点等

**2. 意图分类相似度判断**
- 判断请求类型是否相同（同类型→可复用）
- 结合语义相似度 + 意图分类

**3. 计划结构化转换**
- 非结构化文本 → 结构化执行计划
- 替换关键参数

### 实验配置
- 真实数据集：个人助手请求
- 框架：AutoGen + GPT-4
- 基线：GPTCache等

---

## 5. 核心发现

### 主要结论
1. **有效计划复用率**：93%
2. **相似度F1**：0.9718（+6.8%提升）
3. **相似度准确率**：0.9459（+13.06%提升）
4. **延迟降低**：93.12%

### 关键数据/指标
- 原始请求相似度阈值0.75判断：不相似的请求（"后天合肥到北京" vs "明天长沙到上海"）
- 关键参数：时间、起点、终点
- 延迟：25秒 → 1.7秒（毫秒级额外开销）

### 消融实验
- 验证意图分类方法的有效性
- 验证参数替换准确性

## 6. 创新点

1. **针对计划的语义缓存**：首个专门针对LLM Agent计划复用的机制
2. **意图分类相似度**：超越纯文本相似度，判断请求类型
3. **关键参数识别**：语义级参数提取和替换

## 10. 对Agent实践的指导

### 可直接借鉴的方法
1. **计划缓存**：相同意图请求复用计划
2. **意图分类**：判断请求类型而非纯文本相似
3. **参数化**：关键参数替换实现复用

### 需要适配的场景
- 高频重复请求场景
- 有明确意图分类的任务
- 对延迟敏感的应用

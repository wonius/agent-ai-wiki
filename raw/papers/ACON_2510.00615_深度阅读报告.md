# ACON: Optimizing Context Compression for Long-Horizon LLM Agents

**论文标题**: ACON: Optimizing Context Compression for Long-Horizon LLM Agents
**作者**: Minki Kang (KAIST*), Wei-Ning Chen, Dongge Han, Huseyin A. Inan, Lukas Wutschitz, Yanzhi Chen, Robert Sim, Saravan Rajmohan (Microsoft)
**arXiv**: 2510.00615 | **发表**: 2025-10-01（v2）| **会议**: arXiv preprint

---

## 一句话概括

ACON 通过**自然语言空间的压缩准则优化**，将 LLM Agent 的环境观察和交互历史压缩为简洁但信息完整的摘要，在降低 26-54% 内存占用的同时保持任务性能，且可蒸馏到小模型部署。

---

## 科学问题

### 核心问题

LLM Agent 在长程任务中面临一个**本质张力**：上下文既是决策的基础，又是成本和噪声的来源。随着交互历史不断累积，上下文呈**无界增长**，带来两大挑战：

1. **推理成本**：Transformer 计算成本随上下文长度线性增长，长程任务成本急剧上升
2. **信息稀释**：过长的上下文让模型被过时或不相关信息干扰，有效信号被噪声淹没（Shi et al., 2023）

### 背景与重要性

当前 Web Agent、邮件管理、文档处理等 productivity 场景要求 Agent 协调异构工具并维护精确状态信息。与简单 agent 任务不同，这些环境需要保留多种信号类型：事实历史、动作-结果关系、演化中的环境状态、成功前提条件、未来决策线索。

### 现有方法不足

现有压缩方案各有局限：
- **文档/RAG 压缩**：假设单步推理，不适合多步 agent
- **对话摘要**：适合会话连贯性，不适合跨步骤状态保持
- **KV Cache 压缩**：低级别压缩，与 agent 层面的语义压缩正交
- **近期 agent 压缩方法**（Deng et al., 2023; Lee et al., 2025）：要么依赖朴素 prompting，要么局限于狭窄域，泛化能力差

## 技术路线

### 核心技术：压缩准则优化（Compression Guideline Optimization）

ACON 的核心不是手工设计压缩 prompt，而是**数据驱动优化压缩准则**。给定成对轨迹（full context 成功但 compressed context 失败的案例），用 capable LLM 分析失败原因，并相应地更新压缩准则——整个过程在自然语言空间中进行，**无需梯度更新**，可直接用于闭源或生产模型。

具体流程：
1. 初始压缩准则：手工设计的基础准则
2. 失败轨迹收集：在环境中运行 agent，收集压缩后失败的案例
3. 失败分析：用 LLM 分析失败原因（哪些关键信息被错误压缩了）
4. 准则更新：根据分析结果更新压缩准则文本

### 观察压缩 vs 历史压缩

ACON 同时压缩两类上下文：
- **环境观察压缩（Observation Compression）**：将每步的 tool 输出压缩为关键信息摘要
- **交互历史压缩（History Compression）**：将多步历史压缩为紧凑表示

### 实验设计

验证在三个 benchmark 上进行，每个需要 15+ 交互步骤：
- **AppWorld**（58个真实世界任务，15-50步）
- **OfficeBench**：跨 Office 工具的多步任务
- **Multi-objective QA**：多目标问答

### 蒸馏到小模型

将优化后的 LLM 压缩器蒸馏到更小的模型（如 Qwen-14B），保留 95% 以上精度，大幅降低额外模块的推理开销。

---

## 核心发现

### 主要结论

1. **内存降低 26-54%**：在 AppWorld 上，ACON 将 peak tokens 降低 26-54%，同时保持准确率
2. **蒸馏有效性**：蒸馏后的小模型保留 >95% 的 teacher 精度
3. **小模型提升显著**：通过缓解长上下文干扰，小模型在 AppWorld 上提升 32%，OfficeBench 上 20%，Multi-objective QA 上 46%

### 关键数据

| 模型 | 内存降低 | 准确率变化 |
|------|---------|-----------|
| gpt-4.1 | -26% peak tokens | 持平（+0） |
| gpt-4.1-mini | -24% peak tokens | +18% |
| Qwen3-14B（蒸馏） | -31% peak tokens | +26% |

### 机制解释

ACON 有效的原因：压缩准则优化找到了**任务相关关键信号**和**环境特定信息保留**的平衡点。相比手工 prompt，优化的准则能够保留决策必需的状态信息（email ID、文件版本、API 格式等），丢弃干扰性细节。

---

## 创新点

1. **梯度自由、自然语言空间的压缩准则优化**：无需参数更新，可用于任何 LLM（包括 API 模型）
2. **统一框架**：同时支持环境观察压缩和交互历史压缩，适用于任意 Agent 任务
3. **低成本部署方案**：通过蒸馏将优化后的压缩器部署到小模型，保留 95%+ 精度

---

## 与现有方法对比

| 方法 | 压缩粒度 | 需梯度 | 泛化性 | 适用场景 |
|------|---------|-------|-------|---------|
| ACON（本文） | 语义级（观察+历史） | 否 | 高 | 任意 Agent |
| RAG 压缩 | 文档级 | 否 | 中 | 单步 QA |
| 对话摘要 | 会话级 | 否 | 中 | 对话系统 |
| KV Cache 压缩 | token 级 | 是 | 高 | 底层推理 |
| Deng et al. 2023 | 启发式 | 否 | 低 | 狭窄域 |

---

## 局限性与发展方向

**局限性**：
- 压缩准则优化依赖 capable LLM 进行失败分析，增加了优化成本
- 目前验证在 AppWorld、OfficeBench、Multi-objective QA，需要更多真实世界 benchmark
- 蒸馏过程可能丢失某些环境特定信号

**发展方向**：
- 自动化失败轨迹收集和准则更新循环
- 跨环境迁移：一个环境优化的准则能否泛化到新环境
- 与 KV Cache 压缩的更深层次结合

## 对你研究的意义

作为 AI Agent 研究者，ACON 直接解决了长程任务中的上下文膨胀问题。关键启发：

1. **语义压缩 > 启发式压缩**：手工设计压缩规则效果差，数据驱动的压缩准则优化是正确方向
2. **小模型 + 专项优化可以超越大模型**：46% 的性能提升证明，找到关键信号比模型大小更重要
3. **梯度自由 = 可用于生产**：无需梯度意味着可以直接用于 GPT-4o、Claude 等闭源模型
4. **与 AgentFold 正交**：AgentFold 的主动上下文折叠和 ACON 的压缩准则优化可以互补——一个是"什么时候折叠"，一个是"怎么压缩才不丢失关键信息"
